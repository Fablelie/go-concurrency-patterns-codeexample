# 🚀 Go Concurrency Patterns Blueprint

โปรเจกต์นี้รวบรวมตัวอย่างโครงสร้างและรูปแบบการออกแบบ **Concurrency Patterns** ในภาษา Go ตั้งแต่ระดับพื้นฐานจนถึงระดับสูง (Advanced) เพื่อเป็นแนวทางในการเขียนระบบที่รองรับการทำงานแบบขนานอย่างมีประสิทธิภาพ ปลอดภัย และประหยัดทรัพยากร

---

## 📌 สรุปเปรียบเทียบทั้ง 9 Patterns

### 01. Generator Pattern
* **คำนิยาม:** ฟังก์ชันที่ผลิตข้อมูลส่งออกมาทาง Channel ขาออกแบบ Asynchronous ทันทีที่ข้อมูลแต่ละชิ้นเสร็จ
* **Use Case:** ระบบรันเลขเอกสาร/Order ID อัตโนมัติ, ระบบสตรีมอ่านไฟล์ขนาดใหญ่ทีละบรรทัด
* **🟢 ข้อดี:** ประหยัด Memory ไม่ต้องสร้าง Slice ขนาดใหญมารองรับข้อมูลทั้งหมดในคราวเดียว
* **🔴 ข้อเสีย:** หากฝั่งรับข้อมูลประมวลผลช้าเกินไป อาจทำให้ Goroutine ฝั่งผลิตติดบล็อก (Block) ได้

### 02. Fan-In & Fan-Out Pattern
* **คำนิยาม:** การกระจายงานให้หลาย Worker ช่วยกันทำ (Fan-Out) แล้วรวบรวมผลลัพธ์จากทุกช่องทางกลับมาที่ Channel เดียว (Fan-In)
* **Use Case:** ระบบดึงข้อมูลสภาพอากาศหรือราคาสินค้าจาก Third-party API พร้อมกันหลายแห่งเพื่อนำมาเปรียบเทียบ
* **🟢 ข้อดี:** ประหยัดเวลาทำงานรวมได้อย่างมหาศาล เพราะทุกงานถูกรันขนานพร้อมกันแยกเป็นอิสระ
* **🔴 ข้อเสีย:** การจัดการลอจิกค่อนข้างซับซ้อน ต้องควบคุมจังหวะการปิด Channel (close) ให้ถูกต้องเพื่อป้องกัน Deadlock

### 03. Worker Pool Pattern
* **คำนิยาม:** การจำกัดจำนวน Goroutines (Workers) ที่แน่นอนไว้ล่วงหน้า เพื่อให้รุมช่วยกันดึงงานจากคิวกลางไปทำ
* **Use Case:** ระบบย่อขนาดรูปภาพ (Image Resizing) หรือระบบส่งอีเมลแจกจ่ายลูกค้าจำนวนมาก
* **🟢 ข้อดี:** ควบคุมทรัพยากรระบบ (CPU/Memory) ได้คงที่ ป้องกันไม่ให้แอปพลิเคชันหรือ Database ล่มจากการโดนรุมใช้งาน
* **🔴 ข้อเสีย:** หากกำหนดจำนวน Worker น้อยเกินไป งานในคิวอาจค้างสะสมและระบายไม่ทัน

### 04. Pipeline Pattern
* **คำนิยาม:** การแบ่งขั้นตอนการประมวลผลออกเป็นสถานีย่อย (Stages) เชื่อมต่อกันด้วย Channel เหมือนสายพานโรงงาน
* **Use Case:** ระบบล้างข้อมูลและนำเข้าข้อมูล (Data ETL) เช่น อ่านไฟล์ -> กรองข้อมูลเสีย -> แปลงข้อมูล -> บันทึกลง DB
* **🟢 ข้อดี:** ข้อมูลไหลแบบ Stream (ชิ้นแรกผ่านสเตจแรกแล้วไหลต่อไปสเตจสองทันที) โค้ดแยกสัดส่วนชัดเจน (Decoupled)
* **🔴 ข้อเสีย:** ถ้าสเตจใดสเตจหนึ่งทำงานช้า จะกลายเป็นจุดคอขวด (Bottleneck) ที่ทำให้สายพานทั้งหมดชะลอตัว

### 05. Future / Promise Pattern
* **คำนิยาม:** สั่งทำงานหนักไว้เบื้องหลังทันที และคืนค่าส่งกลับมาเป็น Object สัญญา (Channel) เพื่อให้โค้ดหลักไปทำงานอื่นต่อได้ก่อน
* **Use Case:** หน้าเว็บแดชบอร์ดที่ต้องโหลดข้อมูลโปรไฟล์ผู้ใช้และประวัติการซื้อพร้อมกัน โดยไม่ต้องรอผลลัพธ์ทีละตัว
* **🟢 ข้อดี:** โค้ดหลักไม่ต้องหยุดรอ (Non-blocking) สามารถทำหลายกิจกรรมไปพร้อมกันได้จนกว่าจำเป็นต้องใช้ข้อมูลจริง
* **🔴 ข้อเสีย:** หากเปิดอ่านข้อมูลจาก Future เร็วเกินไป โค้ดจะหยุดรอ (Block) อยู่ดี ทำให้ไม่ได้ประโยชน์เต็มที่

### 06. Rate Limiter Pattern
* **คำนิยาม:** การใช้ตัวควบคุมเวลา (เช่น Ticker) คุมจังหวะการปล่อยงาน เพื่อจำกัดปริมาณการประมวลผลต่อวินาที
* **Use Case:** ระบบยิง API แจ้งเตือนไปยัง LINE Notify หรือระบบ Payment Gateway ที่จำกัดโควตา Request ต่อนาที
* **🟢 ข้อดี:** ป้องกันการถูกบล็อก IP หรือโดนปฏิเสธการให้บริการ (Rate Limit Exceeded) จากระบบปลายทาง
* **🔴 ข้อเสีย:** ทำให้ความเร็วโดยรวมของระบบลดลงตามโควตาที่กำหนดไว้ (ระบบจงใจหน่วงเวลาตัวเอง)

### 07. Pub/Sub (Publish/Subscribe) Pattern
* **คำนิยาม:** ผู้ส่งกระจายข่าว (Publish) เข้าหัวข้อกลาง โดยมีผู้รับ (Subscribers) หลายส่วนแอบฟังและดึงงานไปทำแยกกัน
* **Use Case:** ระบบ Chat Room หรือระบบ Event-Driven เมื่อกดซื้อสินค้าแล้วต้องส่งข้อมูลไปคลังสินค้าและระบบบัญชีพร้อมกัน
* **🟢 ข้อดี:** ระบบแต่ละส่วนทำงานเป็นอิสระต่อกันโดยสมบูรณ์ (Highly Decoupled) เพิ่ม/ลดผู้รับข่าวสารได้ง่ายโดยไม่ต้องแก้โค้ดผู้ส่ง
* **🔴 ข้อเสีย:** ตรวจสอบข้อผิดพลาด (Debug) ลำดับการไหลของข้อมูลได้ยากขึ้นเมื่อระบบขยายตัวใหญ่มาก

### 08. Or-Done Channel Pattern
* **คำนิยาม:** การรวมช่องสัญญาณแจ้งยกเลิกงานจากจุดต่าง ๆ เข้ามาไว้ที่คำสั่งกลางจุดเดียวเพื่อความปลอดภัย
* **Use Case:** ระบบ Microservices ที่ต้องคอยเช็คสัญญาณปิดระบบ (SIGTERM) หรือสัญญาณ Timeout จากฝั่งผู้ใช้ไปพร้อมกัน
* **🟢 ข้อดี:** ป้องกันการเกิดปัญหาหน่วยความจำรั่วไหล (Goroutine Leak) จากการที่มี Goroutine ค้างเติ่งอยู่ในระบบหลังเลิกงาน
* **🔴 ข้อเสีย:** เพิ่มความซับซ้อนในการใช้คำสั่ง select ซ้อนกันหลายชั้นในโค้ด

### 09. Singleflight Pattern
* **คำนิยาม:** การยุบรวมคำสั่งที่ทำงานเหมือนกันในเสี้ยววินาทีเดียวกัน ให้เหลือการทำงานจริงครั้งเดียวแล้วแชร์คำตอบร่วมกัน
* **Use Case:** ระบบป้องกัน Cache Stampede เมื่อคีย์ยอดฮิตบน Cache หมดอายุและมีผู้ใช้รุมกดเข้ามาดูพร้อมกันหลักพันคน
* **🟢 ข้อดี:** ลดภาระการโหลดข้อมูลของ Database หรือ Backend Service ปลายทางได้อย่างมหาศาล
* **🔴 ข้อเสีย:** ข้อมูลที่แชร์ร่วมกันจะเหมือนกันเป๊ะ หากมีความต้องการข้อมูลที่เฉพาะเจาะจงรายบุคคล (เช่น สิทธิ์ผู้ใช้ต่างกัน) จะใช้ไม่ได้

## 🛠️ วิธีการรันโปรเจกต์ (How to Run)

โปรเจกต์นี้รองรับการรันผ่าน PowerShell ด้วยสคริปต์ `run.ps1` ที่อยู่หน้าแรก โดยระบุหมายเลขหรือคำสำคัญของ Pattern ที่ต้องการ:

```powershell
# รันด้วยตัวเลขโฟลเดอร์
.\run.ps1 01

# หรือรันด้วยคำสำคัญ (Keyword)
.\run.ps1 singleflight
```
